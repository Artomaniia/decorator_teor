## Что такое `Decorator`

Паттерн **Decorator** — это способ **добавить объекту новое поведение, не меняя его код напрямую**.

Самая простая мысль:

> мы берем готовый объект, “оборачиваем” его в другой объект и добавляем что-то сверху.

То есть декоратор — это **обертка**.

### Очень бытовая аналогия

Представь обычный кофе.

- Есть **кофе**
- Можно добавить **молоко**
- Можно добавить **сироп**
- Можно добавить **сливки**

Сам кофе не меняется как класс. Но мы можем оборачивать его:

- кофе
- кофе + молоко
- кофе + молоко + сироп
- кофе + сливки + сироп

Это и есть идея `Decorator`:  
**берем базовый объект и постепенно добавляем ему возможности.**

---

# Зачем нужен этот паттерн

Без декоратора часто делают так: создают много наследников.

Например:

- `Coffee`
- `CoffeeWithMilk`
- `CoffeeWithSugar`
- `CoffeeWithMilkAndSugar`
- `CoffeeWithMilkAndSyrup`
- `CoffeeWithMilkAndSugarAndCream`

Проблема очевидна: классов становится слишком много.

`Decorator` решает это красиво:

- есть базовый объект
- есть отдельные декораторы
- их можно комбинировать в любом порядке

---

# Главная идея паттерна

У декоратора обычно есть 3 важные части:

## 1. Базовый интерфейс

Все объекты должны иметь одинаковый набор методов.

Например:

- `get_price()`
- `get_description()`

## 2. Конкретный объект

Это обычный объект, который уже что-то умеет.

Например, просто `Coffee`.

## 3. Декоратор

Это объект, который:

- хранит внутри другой объект
- вызывает его методы
- добавляет сверху свое поведение

---

# Как это выглядит в Python

Сначала покажу **классический паттерн Decorator**, а потом отдельно объясню, почему в Python слово “декоратор” часто еще означает `@something`.

---

# Пример 1. Очень простой и понятный: кофе

## Без декоратора

class Coffee:  
    def get_description(self):  
        return "Обычный кофе"  
  
    def get_price(self):  
        return 100

Работает, но как добавить молоко, сироп, сливки?  
Если через наследование, получится куча классов.

---

## С декоратором

### Базовый компонент

class Beverage:  
    def get_description(self):  
        raise NotImplementedError  
  
    def get_price(self):  
        raise NotImplementedError

Это общий “контракт”: любой напиток должен уметь возвращать описание и цену.

---

### Конкретный объект

class Coffee(Beverage):  
    def get_description(self):  
        return "Обычный кофе"  
  
    def get_price(self):  
        return 100

---

### Базовый декоратор

class BeverageDecorator(Beverage):  
    def __init__(self, beverage):  
        self.beverage = beverage

Он хранит внутри другой напиток.

---

### Декоратор "Молоко"

class MilkDecorator(BeverageDecorator):  
    def get_description(self):  
        return self.beverage.get_description() + " + молоко"  
  
    def get_price(self):  
        return self.beverage.get_price() + 20

---

### Декоратор "Сироп"

class SyrupDecorator(BeverageDecorator):  
    def get_description(self):  
        return self.beverage.get_description() + " + сироп"  
  
    def get_price(self):  
        return self.beverage.get_price() + 15

---

### Использование

coffee = Coffee()  
print(coffee.get_description())  # Обычный кофе  
print(coffee.get_price())        # 100  
  
coffee_with_milk = MilkDecorator(coffee)  
print(coffee_with_milk.get_description())  # Обычный кофе + молоко  
print(coffee_with_milk.get_price())        # 120  
  
coffee_with_milk_and_syrup = SyrupDecorator(coffee_with_milk)  
print(coffee_with_milk_and_syrup.get_description())  # Обычный кофе + молоко + сироп  
print(coffee_with_milk_and_syrup.get_price())        # 135

---

# Что здесь произошло

Вот эта строка:

coffee_with_milk = MilkDecorator(coffee)

значит:

- у нас был объект `coffee`
- мы завернули его в `MilkDecorator`
- теперь новый объект ведет себя как кофе, но с дополнительной логикой

А тут:

coffee_with_milk_and_syrup = SyrupDecorator(coffee_with_milk)

мы обернули уже **декорированный объект** еще одним декоратором.

То есть декораторы можно **наслаивать** друг на друга.

---

# Почему это удобно

Потому что можно собирать поведение как конструктор:

drink = SyrupDecorator(MilkDecorator(Coffee()))

Это читается как:

- взять кофе
- добавить молоко
- потом добавить сироп

# Пример 2. Более “программистский”: уведомления

Представим, есть отправка уведомления.  
Базово оно отправляется просто как текст.

Потом хотим добавлять:

- SMS
- Email
- Telegram
- логирование

## Базовый интерфейс

class Notifier:  
    def send(self, message):  
        raise NotImplementedError

## Простой отправитель

class BasicNotifier(Notifier):  
    def send(self, message):  
        print(f"Отправлено сообщение: {message}")

## Базовый декоратор

class NotifierDecorator(Notifier):  
    def __init__(self, notifier):  
        self.notifier = notifier

## Email-декоратор

class EmailDecorator(NotifierDecorator):  
    def send(self, message):  
        self.notifier.send(message)  
        print(f"Дополнительно отправили Email: {message}")

## SMS-декоратор

class SMSDecorator(NotifierDecorator):  
    def send(self, message):  
        self.notifier.send(message)  
        print(f"Дополнительно отправили SMS: {message}")

## Логирование

class LoggingDecorator(NotifierDecorator):  
    def send(self, message):  
        print("Записываем в лог перед отправкой")  
        self.notifier.send(message)  
        print("Записываем в лог после отправки")

## Использование

notifier = BasicNotifier()  
notifier.send("Привет!")

Результат:

Отправлено сообщение: Привет!

Теперь с декораторами:

smart_notifier = LoggingDecorator(  
    SMSDecorator(  
        EmailDecorator(  
            BasicNotifier()  
        )  
    )  
)  
  
smart_notifier.send("Важное уведомление")

Результат будет примерно таким:

Записываем в лог перед отправкой  
Отправлено сообщение: Важное уведомление  
Дополнительно отправили Email: Важное уведомление  
Дополнительно отправили SMS: Важное уведомление  
Записываем в лог после отправки

---

# Очень важная мысль

Декоратор **не заменяет** основной объект, а **расширяет** его.

То есть основной объект умеет делать базовую вещь,  
а декораторы добавляют новые действия до или после вызова.

---

# Как понять паттерн одной фразой

Вот формулировка, которую можно сказать на паре:

> `Decorator` — это структурный паттерн проектирования, который позволяет динамически добавлять объекту новое поведение, оборачивая его в объект-декоратор с тем же интерфейсом.

Разберем эту фразу:

- **структурный паттерн** — работает со структурой объектов
- **динамически** — можно добавлять поведение во время работы программы
- **оборачивая** — объект помещается внутрь другого объекта
- **с тем же интерфейсом** — снаружи декоратор выглядит как обычный объект

---

# Чем Decorator отличается от наследования

Это частый вопрос.

## Наследование

При наследовании ты заранее создаешь новый класс.

Например:

class MilkCoffee(Coffee):  
    pass

Проблемы:

- комбинаций очень много
- классы быстро размножаются
- сложно гибко комбинировать поведение

## Decorator

С декоратором ты не делаешь миллион классов-комбинаций.  
Ты собираешь поведение из отдельных оберток.

Это гибче.

---

# Когда использовать Decorator

Паттерн полезен, когда:

### 1. Нужно добавлять поведение объектам без изменения их кода

Например, логирование, кэширование, проверку прав, форматирование.

### 2. Нужно комбинировать возможности

Например, можно включить логирование + шифрование + сжатие.

### 3. Наследование слишком разрастается

Если классов становится слишком много, декоратор помогает сократить этот хаос.

---

# Когда он может быть неудобен

Надо честно сказать: паттерн не всегда идеален.

Минусы:

### 1. Появляется много маленьких классов

Если декораторов слишком много, код может стать запутанным.

### 2. Сложнее отлаживать

Потому что объект завернут в объект, тот еще в объект, и так далее.

### 3. Порядок декораторов может влиять на результат

Например:

A(B(obj))

и

B(A(obj))

могут вести себя по-разному.

---

# Теперь самое важное для Python:

# в Python слово “decorator” имеет ДВА смысла

Это очень важно, потому что преподаватель может ожидать один из них.

## Смысл 1. Паттерн проектирования Decorator

Это то, что мы только что разобрали:  
объект-обертка вокруг другого объекта.

## Смысл 2. Декораторы функций в Python

Это конструкция вида:

@my_decorator  
def hello():  
    print("Привет")

В Python это тоже называется “decorator”, но это уже **языковой механизм**, а не только классический GoF-паттерн.

Идея у них похожая:  
**мы оборачиваем поведение и добавляем что-то сверху.**

---

# Декораторы функций в Python — простое объяснение

Сначала без `@`.

def hello():  
    print("Привет")  
  
def decorator(func):  
    def wrapper():  
        print("До вызова функции")  
        func()  
        print("После вызова функции")  
    return wrapper  
  
hello = decorator(hello)  
hello()

Результат:

До вызова функции  
Привет  
После вызова функции

Что произошло:

- была функция `hello`
- мы передали ее в `decorator`
- получили новую функцию `wrapper`
- внутри `wrapper` вызывается исходная функция плюс дополнительная логика

Это очень похоже на паттерн Decorator.

---

## То же самое через `@`

def decorator(func):  
    def wrapper():  
        print("До вызова функции")  
        func()  
        print("После вызова функции")  
    return wrapper  
  
@decorator  
def hello():  
    print("Привет")  
  
hello()

Это просто более красивый способ записать:

hello = decorator(hello)

---

# Пример полезного функционального декоратора

## Логирование вызова функции

def log_decorator(func):  
    def wrapper():  
        print(f"Функция {func.__name__} начала работу")  
        func()  
        print(f"Функция {func.__name__} завершила работу")  
    return wrapper  
  
@log_decorator  
def say_hi():  
    print("Привет!")  
  
say_hi()

Результат:

Функция say_hi начала работу  
Привет!  
Функция say_hi завершила работу

---

# Пример с аргументами

Часто функции принимают параметры, поэтому декоратор делают так:

def log_decorator(func):  
    def wrapper(*args, **kwargs):  
        print(f"Вызываем {func.__name__}")  
        result = func(*args, **kwargs)  
        print(f"{func.__name__} завершена")  
        return result  
    return wrapper  
  
@log_decorator  
def add(a, b):  
    return a + b  
  
print(add(2, 3))

Результат:

Вызываем add  
add завершена  
5

# Зачем нужны декораторы функций в Python

Они часто используются для:

- логирования
- проверки прав доступа
- кэширования
- измерения времени работы
- проверки аргументов
- повторного использования общей логики

---

# Пример: измерение времени выполнения

import time  
  
def timer(func):  
    def wrapper(*args, **kwargs):  
        start = time.time()  
        result = func(*args, **kwargs)  
        end = time.time()  
        print(f"Функция {func.__name__} работала {end - start:.6f} секунд")  
        return result  
    return wrapper  
  
@timer  
def slow_function():  
    time.sleep(1)  
    print("Функция закончила работу")  
  
slow_function()

---

# Пример: проверка доступа

def require_admin(func):  
    def wrapper(user_role):  
        if user_role != "admin":  
            print("Доступ запрещен")  
            return  
        return func(user_role)  
    return wrapper  
  
@require_admin  
def open_settings(user_role):  
    print("Настройки открыты")  
  
open_settings("user")  
open_settings("admin")

Результат:

Доступ запрещен  
Настройки открыты

# Связь между паттерном и `@decorator`

Можно сказать так:

> классический паттерн Decorator работает с объектами,  
> а Python-декораторы чаще работают с функциями и методами,  
> но идея у них одна — добавить поведение, не меняя исходный код напрямую.

Это очень хорошая формулировка для ответа.

---

# Что чаще спрашивают на паре

Вот самые вероятные вопросы и готовые ответы.

## “Почему нельзя просто использовать наследование?”

Потому что наследование создает много классов для всех комбинаций поведения, а `Decorator` позволяет гибко добавлять возможности через обертки.

## “В чем плюс Decorator?”

Он позволяет расширять поведение объекта без изменения его класса и без взрыва количества подклассов.

## “Почему это называется декоратор?”

Потому что объект как бы “украшается” новым поведением.

## “Это динамическое добавление поведения?”

Да. Мы можем во время выполнения оборачивать объект нужными декораторами.

## “Есть ли такое в реальном Python?”

Да. Помимо паттерна проектирования, в Python есть встроенная идея декораторов функций через `@`.

---

# Частая ошибка в понимании

Многие думают, что декоратор просто “что-то вызывает до и после”.

Это не совсем так.

Главная суть не в том, что он делает что-то “до/после”,  
а в том, что он **оборачивает** объект или функцию и **расширяет поведение**, сохраняя исходную сущность.

---

# Как рассказать это простыми словами на паре

Вот версия совсем простым языком:

> Декоратор — это способ добавить объекту новые возможности через обертку.  
> Вместо того чтобы менять сам объект или создавать много наследников, мы помещаем его в другой объект, который добавляет нужное поведение.  
> В Python эта идея встречается и как паттерн проектирования, и как декораторы функций с синтаксисом `@`.

---

# Готовый мини-ответ на 1–2 минуты

Можешь почти дословно сказать так:

> Паттерн Decorator — это структурный паттерн проектирования, который позволяет добавлять объекту новое поведение динамически, не изменяя его исходный класс.  
> Суть в том, что мы не переписываем объект и не создаем множество подклассов, а оборачиваем его в специальный объект-декоратор.  
> Декоратор имеет тот же интерфейс, что и исходный объект, поэтому снаружи им можно пользоваться так же.  
> Например, есть обычный кофе, а декораторы могут добавить молоко, сироп или сливки. Мы можем комбинировать эти декораторы между собой.  
> В Python эта идея особенно важна, потому что кроме паттерна Decorator есть еще декораторы функций с синтаксисом `@`, которые тоже оборачивают функцию и добавляют ей новое поведение, например логирование, проверку доступа или измерение времени работы.
